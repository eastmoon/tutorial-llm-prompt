# 判斷句

大語言模型沒有原生的 if/else 執行引擎，判斷句的本質是：用自然語言或結構標籤，讓模型「理解條件語境」後選擇對應行為。

關鍵是讓條件、分支、執行動作三者的邊界都清晰可辨。

## 格式

如同[變數](./syntax.variable.md)一樣，大語言模型有兩種格式句型，XML 格式是給模型讀的結構，Markdown 格式是給人讀的排版。

由於 Markdown 存在具有模糊性，為避免大語言模型解讀錯誤，應注意以下使用時機：

Markdown 判斷句適合：
✅ 條件少於 3 個，無巢狀需求
✅ 提示詞以人類閱讀為主，偶爾給模型參考
✅ 快速原型，不追求嚴謹執行精度
✅ 受眾習慣 Markdown，XML 反而造成閱讀負擔

XML 判斷句適合：
✅ 條件超過 3 個或有巢狀結構
✅ 需要精確的 else 歸屬
✅ 提示詞由程式動態生成
✅ 對模型執行精度有高要求的生產環境

## 自然語言判斷句

以下為自然語言判斷句範本，這樣的寫法簡單，但條件邊界模糊，多條件時容易產生歧義。

```
如果使用者的問題與程式碼有關，請提供可執行的範例。
如果問題是概念性的，請用比喻說明，不需要程式碼。
如果問題不在你的能力範圍，請明確說明無法回答。
```

## 結構化判斷句

未避免自然語言條件句的邊界模糊問題，建議強化語法為結構化判斷句：

### if/else

if/else 是最常見的假如否則例句。

Markdown 格式
```
## Rules

- **若** 使用者提供程式碼：分析錯誤並給出修正版本
- **若** 使用者只描述問題：先詢問技術堆疊再回答
- **否則**：請使用者澄清問題
```

XML 格式
```
<if condition="使用者提供程式碼">
  <then>分析錯誤並給出修正版本</then>
</if>
```

### Nested

if / else if / else 為典型的巢狀判斷句，常用於多判斷條件的結構句。

Markdwon 格式
```
## Decision Logic

- **若** `USER_LEVEL == 初學者`
  - 使用比喻和生活化例子
  - 避免術語，若必須使用請附解釋
  - **若** 問題涉及程式碼
    - 提供完整可執行範例並加上逐行註解

- **若** `USER_LEVEL == 進階`
  - 直接進入技術細節，省略基礎說明
  - **若** 問題涉及程式碼
    - 提供精簡範例，聚焦關鍵邏輯

- **否則**
  - 以中級程度回答，結尾詢問是否需要調整深度
```

XML 格式
```
<decision_tree>

  <if condition="USER_LEVEL == 初學者">
    <then>
      使用比喻和生活化例子說明。
      避免術語，若必須使用請附上解釋。

      <if condition="問題涉及程式碼">
        <then>提供完整可執行範例，加上逐行註解。</then>
      </if>
    </then>

  <elif condition="USER_LEVEL == 進階">
    <then>
      直接進入技術細節，省略基礎說明。
      可使用專業術語，不需解釋。

      <if condition="問題涉及程式碼">
        <then>提供精簡範例，聚焦在關鍵邏輯。</then>
      </if>
    </then>
  </elif>

  <else>
    預設以中級程度回答，並在結尾詢問是否需要調整深度。
  </else>

</decision_tree>
```

巢狀判斷也是 Markdwon 與 XML 格式最大的差異句型，其中差異如下表：

| 比較點 | Markdown | XML |
| :- | :-- | :-- |
| 巢狀邊界 | 靠縮排，模型可能誤判層級| 開閉標籤強制定義，無歧義 |
| 條件與動作的分離 | 混在同一行，語義依賴措辭 | condition 屬性與 <then> 明確分離 |
| else 的歸屬 | 不清楚 else 對應哪個 if | 結構上明確對應 |
| 模型解讀| 視為「建議段落」| 視為「結構化指令」|

### Switch

Switch 常用於單一變數多種可能例句。

Markdwon 格式
```
## Response Style

根據 `$REQUEST_TYPE` 選擇對應方式：

| $REQUEST_TYPE | 執行方式 |
|--------------|---------|
| 解釋概念 | 一句定義 → 展開說明 → 舉例 |
| 除錯 | 重述問題 → 分析原因 → 修正方案 |
| 比較 | 表格呈現差異 → 情境化建議 |
| 撰寫文件 | 目的 → 參數 → 範例 → 注意事項 |

若 `$REQUEST_TYPE` 不在以上範圍，詢問使用者希望以哪種方式獲得協助。
```

XML 格式
```
<switch variable="$REQUEST_TYPE">

  <case value="解釋概念">
    先給一句話定義，再展開說明，最後舉一個實際例子。
  </case>

  <case value="除錯">
    先重述問題確認理解，逐步分析可能原因，提供修正方案。
  </case>

  <case value="比較">
    以表格呈現差異，最後給出情境化的選用建議。
  </case>

  <case value="撰寫文件">
    使用正式語氣，結構包含：目的、參數說明、範例、注意事項。
  </case>

  <default>
    詢問使用者希望以哪種方式獲得協助。
  </default>

</switch>
```

### Guard Clause

Guard Clause 是一種優先判斷不合法的用法。

Markdown 格式
```
## ⛔ 前置條件（不符合則停止，不執行後續規則）

1. 若問題涉及個人隱私資料 → 拒絕回答並說明原因，**停止**
2. 若問題語言非繁體中文或英文 → 請使用者改用正確語言，**停止**
3. 若問題少於 5 字且語意不明 → 請使用者補充背景，**停止**

---

## ✅ 主要邏輯（通過前置條件後執行）

正常回答流程...
```

XML 格式
```
<guards>
  <!-- 前置條件：不符合則終止並回應原因 -->

  <guard condition="問題涉及真實個人隱私資料">
    <action>拒絕回答，說明隱私保護原則，不繼續執行後續規則。</action>
  </guard>

  <guard condition="問題語言非繁體中文且非英文">
    <action>請使用者改用繁體中文或英文提問，不繼續執行後續規則。</action>
  </guard>

  <guard condition="問題字數少於 5 字且語意不明">
    <action>請使用者提供更多背景資訊，不繼續執行後續規則。</action>
  </guard>

</guards>

<!-- guards 全部通過後，才執行以下主要邏輯 -->
<main_logic>
  正常回答流程...
</main_logic>
```


## 強化語（Anchoring Words）

對大語言模型，僅是結構還是有可能被忽略或模糊處理，需加上明確的執行語氣：

```
❌ 弱語氣（模型可能選擇性忽略）
如果有程式碼的話，可以提供範例。

✅ 強語氣（明確要求執行）
當偵測到使用者提供程式碼時，必須提供修正後的完整程式碼，不得省略任何行。

✅ 加上否定約束
若問題屬於概念解釋類型，禁止主動提供程式碼範例，除非使用者明確要求。
```
